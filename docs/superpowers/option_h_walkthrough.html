2. Full strategy lifecycle — 6 steps

A concrete walkthrough of adding one new strategy from blank Pine editor to live paper trading. Approximate time budget next to each step.

1
Author in Pine Script
Where: TradingView web editor  ·  Time: ~1h (varies by strategy complexity)
Write the strategy logic in Pine v6. Use strategy.entry / strategy.exit with descriptive id / comment strings — these travel to both the trade list CSV and the live alert payload. Add an alert_message JSON body that the webhook receiver can parse directly (symbol, side, quantity hint, strategy tag).
2
Backtest in TradingView Strategy Tester
Where: TradingView chart  ·  Time: ~10 min per symbol/window
Load the Pine strategy on the target symbol. Set the date range. Click the "List of Trades" tab. Visually eyeball the trades and TradingView's built-in metrics (net P&L, PF, max DD, run-up). If the strategy is clearly broken, iterate in Pine before going further — no need to burn tradelab cycles on obvious losers.
3
Export trade list as CSV
Where: Strategy Tester ·  Time: 30 seconds
Click the export icon in the List of Trades tab, choose CSV. File saves to Downloads with every closed trade (entry/exit time + price, position size, signal tag, run-up, drawdown, P&L). This is your ground truth — exactly what TradingView saw.
4
Score in tradelab
Where: Windows terminal  ·  Time: ~2 min
Run tradelab score-from-trades my_strategy.csv --universe my_universe. The CSV adapter converts TradingView's format to tradelab's internal trade-record schema, then runs the full robustness suite (DSR, LOSO, Monte Carlo, regime spread, etc.) on the ledger. Verdict: ROBUST / INCONCLUSIVE / FRAGILE plus a full signal breakdown.
5
Review verdict on dashboard
Where: http://localhost:8877 Research tab  ·  Time: ~10 min
The new strategy appears as a row in the pipeline table. Heat colors, verdict pill, per-signal breakdown. If FRAGILE — iterate in Pine (Step 1), re-export (Step 3), re-score (Step 4). If ROBUST — proceed.
6
Deploy live (paper)
Where: TradingView alert dialog  ·  Time: ~5 min
In TradingView, add an alert on the strategy's strategy.entry / strategy.exit events. Paste your webhook URL (your Windows box + ngrok / tunnel). Pine fires alerts on every live bar — the FastAPI receiver catches them, validates, and submits the order to Alpaca paper. Dashboard's Live Strategies card shows the position.
Per-strategy ongoing cost after step 6: roughly 15 min. When you change a Pine parameter, re-run Steps 3–5 (export → score → review). The Pine alert keeps running; no code is touched on the Windows side.

Leave-One-Symbol-Out (LOSO)PARTIALNeeds per-symbol CSVs. Run the Pine strategy on each symbol separately in TV, export each, tradelab recombines. ~N times the export effort where N = symbol count.
Walk-forward validationPARTIALApproximate by running Pine over non-overlapping rolling windows (e.g., 2020, 2021, 2022 separately). Manual but doable. No automated IS/OOS split.

Automated parameter optimization (Optuna)LOSTNeeds the code. No way to feed params to a TradingView strategy programmatically.
What-If parameter slidersLOSTSame reason — needs a callable Python backtest.

Entry-delay robustness stress testLOSTRequires shifting entries programmatically. Can be approximated by manually editing Pine's entry bar logic and re-exporting, but it's tedious.
Noise-injection stress testLOSTRequires injecting synthetic OHLC noise — Pine can't do that to itself.

5. Risks & mitigations

R1 — TradingView changes CSV export format Has been stable since ~2023. Still, TV ships UI changes frequently. Mitigation: CSV adapter includes a schema-version check that fails loudly; downgrade to "paste trade list into the dashboard" manual fallback while schema updates.
R2 — Webhook receiver downtime = missed trades If the receiver crashes at 09:32, any alerts between then and restart are silently dropped. Mitigation: run as Windows service with auto-restart; a watchdog that compares "expected positions from last hour of alerts" vs "actual Alpaca positions" and alarms on divergence. TradersPost warns about the same failure mode; we at least control ours.
R3 — TradingView alert IP ranges change unannounced TV has updated webhook source IPs twice in the last 18 months. Firewall rules fail silently. Mitigation: receiver validates a shared-secret in the alert JSON body, not IP. Works regardless of source IP.
R4 — Pine backtest ≠ live behavior (fill-model drift) TV's Strategy Tester fills at bar close; live alerts fire in real time. Outcomes can differ. Mitigation: run first real strategy with tiny paper position sizes; compare live trade ledger vs backtest ledger over 2 weeks; flag if PF delta > 20%. Same risk exists under every option that uses TV — not unique to H.
R5 — Complex Pine features (multi-symbol, bracket orders) don't export cleanly For the ~30% of strategies that use request.security or pyramiding, the CSV loses context. Mitigation: fall back to manual Python translation (Option A) for those specific strategies. Document which Pine features trigger the fallback in the runbook.
