Rationale appendix
Why these specific changes — the alpha argument
The 4 framing flaws this fixes:
  1. Verdict was one-shot. Now: every Live card shows verdict freshness; calibration banner triggers re-run prompts when accepted cards drift.
  2. Zero regime context. Now: regime banner up top; per-regime PF in the Score modal; regime-fit column on Pipeline.
  3. Pipeline & Live didn't talk. Now: candidates anchored to live book on Pipeline (Corr column) AND in Score modal (Relative context, Portfolio Fit).
  4. Sig modal cells had no anchor. Now: Relative context section under diagnostics: every metric compared to live-card median + worst.
The 3 NEW alpha plays:
  1. Regime conditioning — banner + per-regime PF + regime-fit column. Enables strategy rotation. Single biggest play.
  2. DD correlation + entry-time overlap — Portfolio Health panel + extra columns in Portfolio Fit. Catches silent diversification failures.
  3. Strategy decay sparkline + K-S divergence — replaces "rolling PF tracking error" with two more rigorous signals on each Live card.
Plus your gameplan, integrated:
  • Hold-out as gate (top of Score modal + Pipeline column), not signal #10
  • Correlation pre-Accept on Pipeline + hard gate in Score modal
  • Override logging panel
  • Tracking error preserved as TE bar — but now feeds back into Calibration banner
What I deliberately did NOT add
  • PnL attribution panel. Useful for quants but the math is heavy and the value is "understanding" not "deciding." Defer.
  • Capacity/friction sensitivity chart. At retail size this is a sanity check, not a daily driver. Add only if you scale beyond ~$2M.
  • Synthetic-series bootstrap. Better than the existing MC for overfit detection, but it's a backend signal change — would just be one more cell in the diagnostics grid, not a UI feature. Worth doing but not a Research-tab redesign item.
  • Always-visible Robustness panel. Your existing Sig modal pattern is correct — modal-on-demand. Keep.
  • Compare-N-runs view. Already in your v1.5 backlog. No reason to surface it here.
Build order if you accept this — by dependency, not by visibility
  1. Slice 1a — Hold-out as a gate. Already in your gameplan. Render it as a separate Pipeline column + dedicated Score modal section (not signal #10). Half day.
  2. Slice 1b — Relative context section. Pure frontend; reads existing live-card metrics. Adds anchor to every cell. ~half day.
  3. Slice 2 — Correlation gate (return + DD + entry-time, all three at once). Persist returns AND drawdown series AND entry timestamps at Accept. Compute all three pairwise. Render in Portfolio Fit panel + Pipeline Corr column. Slightly bigger than your 1-day estimate (~1.5 days) but DD corr and entry-time are cheap once the persistence + Pearson loop is built.
  4. Slice 3 — Regime detection module. New module tradelab/regime.py. Classifies bars by vol/trend/breadth. Bucketed PF computed offline at run time, persisted with backtest_result.json. Renders as Regime banner + per-regime table + regime-fit column. ~2 days.
  5. Slice 4 — Decay sparkline + K-S divergence. Replaces planned tracking-error work. Same "live trade log → rolling stat" plumbing, just two stats instead of one. ~1.5 days.
  6. Slice 5 — Calibration meta-stat banner. Reads slice-4 outputs across last N accepted cards. ~half day if 4 is in.
Net: ~6 days vs your current ~5-day plan, but covers regime + DD corr + decay + calibration loop on top.
